# LANEs_CEO 多智能体秘书中枢 V1 需求规格

## 1. 项目背景

Lane 希望建设一套覆盖个人学习、科研、信息获取、日程管理和复盘总结的多智能体秘书助理团队。该系统不是单一聊天机器人，而是一个由本地常驻总控、消息入口、角色工作流、定时任务、权限能力和质量治理机制组成的个人助理中枢。

系统总控 agent 命名为 `LANEs_CEO`。它作为 Lane 的主技术顾问和任务分发中枢，在授权范围内调用本机技能库、MCP、CLI、hook、文档、应用和外部网站能力，接收 Lane 从飞书、微信、QQ 下发的任务，协调 actor/critic 子角色完成工作，并将结果、风险和待确认事项回传给 Lane。

V1 要求将当前已定义的 14 个子角色全部纳入设计，并保证它们在常驻运行、定时触发、质量审核、评分记录、文件落盘和通知回传上遵循统一规范。

## 2. V1 目标与成功标准

### 2.1 核心目标

V1 必须完成以下目标：

1. 建立一个本地常驻的 `LANEs_CEO` 中枢，统一接收飞书、微信、QQ 的任务入口。
2. 将外部指令和定时任务归一化为可持久化、可追踪、可调度的作业。
3. 以 `actor -> critic -> LANEs_CEO -> Lane` 为默认交付链路，保证重要结果在发送前经过审核。
4. 交付当前定义的论文写作、论文调研、AI 新闻、GitHub 热点、邮箱摘要、日报、反思总结等角色工作流。
5. 记录 actor 和 critic 的任务评分，支持月度评级，并将可查看报表同步到 `G:\blog\claude_code_useage\multi-agent\Fraction.xlsx`。
6. 在高自治执行的前提下，为凭据、验证码、敏感消息、外发通知和文件写入建立明确的安全边界。

### 2.2 首轮成功标准

V1 的成功不是单次演示成功，而是能够连续日常运行。首轮验收建议以连续 7 天观察窗口为准，并满足以下条件：

- 定时任务按 `Asia/Shanghai` 时区触发，执行状态可查询。
- 飞书、微信、QQ 的 Lane 指令都能进入统一任务链路。
- actor 产物在 critic 未放行前不会直接提交给 Lane。
- 失败任务能保留上下文、记录失败原因、按规则重试或通知。
- 评分记录不会因并发任务、重复触发或报表同步而损坏。
- 指定文件产物可落盘，通知可回传，敏感信息不会进入普通日志或产物正文。

## 3. V1 范围

### 3.1 V1 必做

V1 必须覆盖：

- `LANEs_CEO` 本地常驻总控能力。
- 飞书、微信、QQ 三类消息入口。
- 当前已定义的 14 个 actor/critic 角色。
- 任务持久化、角色调度、定时触发、结果通知、失败追踪。
- critic 审核放行、任务评分、月度评级、Excel 报表输出。
- 邮箱登录与摘要、论文与资讯搜集、日报询问、反思总结落盘。
- DeepSeek 余额检查和低余额提醒。
- 现有 `secret.md` 的兼容读取以及后续凭据迁移要求。

### 3.2 V1 暂缓

以下内容不作为 V1 的交付前提：

- 尚未在本文档定义的旅游、餐饮、出行等新业务角色。
- 完整的多用户权限体系。V1 默认服务 Lane 本人。
- 对所有第三方平台都提供专门官方集成。V1 可按 CLI、MCP、网页登录等可用能力适配。
- 自动修改 actor/critic 模型本体或自动替换智能体实现。V1 先记录评分和评级结果，再由后续机制决定晋升、鼓励、敦促或替换。

### 3.3 明确边界

- `LANEs_CEO` 可以在授权范围内高自治执行任务，但不得把明文密码、验证码、密钥、Cookie、访问令牌写入普通日志、通知消息、论文笔记、日报或反思总结。
- 任务自动化不得绕过 Lane 明确要求的人工确认事项。例如手机验证码、论文录用/拒稿邮件确认、 critic 指出的高风险事项。
- 所有角色默认通过总控调度，不应主动抢占其他角色的工作目录、配置文件、凭据上下文或输出文件。

## 4. 总体运行设计

### 4.1 `LANEs_CEO` 职责

`LANEs_CEO` 是 V1 的本地常驻中枢，至少负责：

- 接收飞书、微信、QQ 消息和系统定时触发。
- 校验任务来源，默认仅接受已授权 Lane 身份发起的高权限任务。
- 将原始指令解析为统一任务请求，并分配给对应 actor/critic 角色组。
- 为每个作业生成唯一 ID，保存作业状态、输入、产物、失败原因和通知记录。
- 根据权限和安全规则调用技能库、MCP、CLI、浏览器、文档、应用和本地文件能力。
- 管理 actor 与 critic 的审核回路，只有 critic 放行的产物才能正式回传给 Lane。
- 维护评分、月度评级、报表同步和异常升级。

### 4.2 默认交付链路

每个业务作业默认遵循以下链路：

1. `LANEs_CEO` 接收任务或定时触发，并创建作业。
2. actor 获取输入、执行任务、生成结构化产物。
3. critic 审查 actor 产物，输出审核结论、评分、问题和是否放行。
4. 未放行时，作业回流给 actor 修订，并保留 critic 反馈。
5. 放行后，`LANEs_CEO` 汇总结果、风险和待确认事项，发送给 Lane 或目标群组。
6. 作业完成后写入运行记录、评分记录和必要产物索引。

### 4.3 角色隔离与并发

V1 必须避免多角色互相干扰：

- 每个作业应有独立作业上下文，包含输入、角色、权限范围、工作目录、产物路径和状态。
- 涉及同一文件、同一账号会话或同一报表时，应采用排队、锁或幂等检查，避免冲突写入。
- 角色配置和临时文件应分区存放，不得以覆盖共享配置的方式切换角色。
- 定时任务和手动任务同时触发时，必须通过作业 ID、触发来源和幂等键区分。

## 5. 权限、安全与登录规则

### 5.1 高自治执行原则

V1 默认采用高自治策略：

- 对低风险查询、文档生成、文件读取、定时汇总、受控目录写入和通知回传可自动执行。
- 对已在任务目标中明确要求的网页登录、邮箱摘要、论文下载、标记普通邮件已读等动作可自动执行。
- 对系统无法确认后果的高风险动作，应将风险说明和待确认事项回传给 Lane。

### 5.2 登录与授权优先级

当任务需要授权或登录时，默认顺序为：

1. 优先使用已有 CLI、MCP、官方 API 或已授权应用连接。
2. 如果无可用命令行或连接能力，再尝试浏览器登录授权。
3. 如需参考网页登录授权流程，可参考 `G:\blog\claude_code_useage\Lecture-Reserve-SEU-master` 的已有做法。
4. 如果需要邮箱验证码，可在授权范围内处理。
5. 如果需要手机验证码，必须通过飞书向 Lane 请求验证码或确认。

### 5.3 凭据管理

- V1 兼容读取 `G:\blog\claude_code_useage\multi-agent\secret.md` 中现有凭据。
- 后续实现必须为凭据迁移预留接口，目标是迁移到更安全的凭据存储方式，例如系统密钥链、加密凭据库或受控环境变量注入。
- 凭据读取应按任务最小可用范围进行，避免把不相关账号凭据暴露给不相关角色。
- 不得把 `secret.md` 内容复制到评分报表、日志、通知、作业摘要或角色产物中。

### 5.4 审计与通知

以下事件应保留审计记录或通知记录：

- 任务已接收、已开始和已完成的飞书状态反馈。
- 登录失败、验证码缺失、凭据缺失。
- 外部消息发送失败。
- 任务超时、连续重试失败、重复触发被抑制。
- critic 连续驳回导致任务无法交付。
- 涉及高风险邮件、敏感外发或余额告警的任务。

## 6. 评分、审核与月度评级

### 6.1 评分对象

- critic 负责对对应 actor 的任务完成情况评分。
- `LANEs_CEO` 负责对 critic 的审核质量评分。
- 论文写作 `LANE-actor` 与论文写作审核 `LANE-critic` 可不纳入月度淘汰评级，因为其结果与 Lane 提供的初始文档、实验数据和模板质量强相关。
- 除上述特殊角色外，其余角色均应纳入评分与月度评级。

### 6.2 分数与评级

单次评分满分为 100 分。为避免区间歧义，V1 采用以下评级区间：

| 分数区间 | 评级 | 月度处理 |
| --- | --- | --- |
| 90-100 | 优秀 | 继续保持 |
| 80-89 | 良好 | 鼓励并记录改进点 |
| 70-79 | 合格 | 敦促改进 |
| 0-69 | 不合格 | 进入替换或重构候选 |

月度评级应基于当月任务评分汇总得出，并记录评价依据，而不是只保存一个最终标签。

### 6.3 critic 放行规则

- critic 输出必须包含审核结论、评分、问题清单、是否放行和是否需要回流 actor。
- 默认只有 critic 明确放行后，`LANEs_CEO` 才能把正式产物提交给 Lane。
- 未放行不等于任务丢弃。系统应保留反馈、进入修订或失败升级链路。
- 论文写作审核的标准更严格：论文写作 `LANE-critic` 需要检查模板格式、全文框架和投稿质量要求；只有达到其明确放行标准后，才可提交给 `LANEs_CEO` 通知 Lane 审查。

### 6.4 评分落地

- 运行态评分、评级、审核反馈应先写入本地结构化存储。
- `Fraction.xlsx` 是 Lane 查看和归档的报表输出，不应成为唯一运行态账本。
- 报表至少应能展示任务时间、角色、评分、评级、critic 结论、问题摘要、月度汇总和处理建议。

## 7. 关键运行规则

### 7.1 时间与定时任务

- V1 默认时区为 `Asia/Shanghai`。
- 每周一 `09:00` 触发论文调研、GitHub 热点和 AI 新闻工作流。
- 每天 `09:00` 触发邮箱摘要工作流。
- 每天 `09:10` 触发日报工作流。
- 每天 `23:00` 触发反思总结工作流。
- 定时任务重复触发时，应依据任务类型、日期和触发来源进行幂等判断，避免同一周期产物重复覆盖。

### 7.2 文件产物与冲突

- 论文调研下载目录为 `G:\blog\claude_code_useage\multi-agent\paper-find`，每个周期按具体日期建立子目录。
- 反思总结默认写入 `G:\blog\claude_code_useage\multi-agent\conclusion`。
- 产物文件已存在时，不得静默覆盖重要结果。应采用版本号、时间戳、幂等复用或显式冲突记录策略。
- 论文 PDF 重命名应保留可读性，命名目标为“创新点 + 论文标题 + 发表期刊或会议”。

### 7.3 邮箱已读边界

- 邮箱摘要工作流可将普通已处理邮件标记为已读。
- 与论文拒稿、录用、修改意见、审稿决定相关的邮件必须保留未读状态，交由 Lane 确认。
- 这类高优先级邮件应在飞书中额外提醒，并在摘要中说明来源邮箱、主题、时间和需要 Lane 确认的原因。

### 7.4 DeepSeek 余额提醒

- `LANEs_CEO` 需要检查 DeepSeek 用量页面 `https://platform.deepseek.com/usage`。
- 当充值余额小于 `0.5` 时，应在飞书秘书群中额外提醒 Lane 三次。
- 余额检查失败时，应记录失败原因，并按通知规则提示 Lane 无法确认余额状态。

## 8. 角色契约

### 8.1 通用角色契约

每组 actor/critic 角色都必须明确以下信息：

| 字段 | 要求 |
| --- | --- |
| 触发方式 | 手动指令、定时任务或两者兼有 |
| 输入来源 | 用户消息、附件、模板、关注领域文件、邮箱、外部网站等 |
| 可用工具 | 允许调用的技能库、MCP、CLI、浏览器、文档与本地资源范围 |

| 投递位置 | Lane 私聊、飞书秘书群、飞书文档、本地目录或报表 |
| critic 审核项 | 完整性、准确性、来源质量、格式、风险、是否满足 Lane 核心要求 |
| 评分项 | 单次评分、问题摘要、改进建议、是否放行 |
| 失败升级 | 登录失败、外部数据不足、超时、连续驳回和人工确认路径 |

## 9. 最小数据契约

每个任务请求至少包含：

| 字段 | 说明 |
| --- | --- |
| `source_channel` | 来源渠道，例如飞书、微信、QQ、scheduler |
| `sender` | 发送者身份 |
| `raw_message` | 原始消息或触发说明 |
| `task_intent` | 解析后的任务意图 |

## 10. 失败处理与恢复

V1 至少需要定义以下失败处理：

| 场景 | 处理要求 |
| --- | --- |
| 登录失败 | 记录失败原因，尝试可用替代登录方式，必要时通知 Lane |
| 凭据缺失 | 不猜测凭据，不把缺失信息写成成功结果 |
| 手机验证码缺失 | 通过飞书请求 Lane 处理 |
| 外部站点不可达 | 保留作业状态和部分结果，记录来源失败 |
| 消息发送失败 | 记录通知失败，按通知重试策略处理 |
| 任务超时 | 保留中间产物，标记超时并通知 |
| 文件冲突 | 不静默覆盖，采用版本化或冲突记录 |
| 重复触发 | 做幂等判断，避免重复投递或重复覆盖 |
| critic 连续驳回 | 保留问题反馈，升级给 `LANEs_CEO`，必要时请 Lane 决策 |
| DeepSeek 余额无法读取 | 记录无法确认余额状态并通知 |

## 11. V1 验收场景

V1 至少通过以下验收：

1. 飞书、微信、QQ 三类入口都能将 Lane 指令归一化为统一任务请求。
2. actor 产物未被 critic 放行时，不会直接提交给 Lane。
3. 每周论文调研、GitHub 热点、AI 新闻任务能按计划触发并生成规定产物。
4. 每日邮箱摘要、日报、反思总结能按指定时间运行并落到指定位置。
5. 登录失败、验证码缺失、外部站点不可达时，系统能通知并保留任务状态。
6. 邮箱任务遇到论文录用、拒稿、审稿决定或修改意见邮件时，保留未读并额外提醒。
7. DeepSeek 余额低于阈值时，飞书秘书群收到三次提醒。
8. 多任务同时写评分时，运行态评分记录不损坏，`Fraction.xlsx` 可正确生成。
9. 文件重名、重复触发、critic 连续驳回时，都有可追踪处理结果。
10. 凭据读取与迁移要求不会把明文秘密扩散到日志、通知或产物中。

## 12. 后续演进方向

在 V1 连续稳定运行后，可继续扩展：

- 旅游、餐饮、出行、生活服务等新增角色组。
- 更细粒度的权限审批、审计面板和人工确认队列。
- 凭据全面迁移和统一账号会话管理。
- 角色能力版本管理、月度自动复盘、替换候选生成和长期质量趋势分析。
- 对不同消息入口、飞书文档、表格与通知系统的更深集成。
