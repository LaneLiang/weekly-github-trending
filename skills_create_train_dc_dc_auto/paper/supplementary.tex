\section{ODE Solver Details}

The circuit environment uses forward Euler integration with $N=50$ sub-steps
per switching cycle. The solver operates at an effective time step of:

\begin{equation}
\Delta t = \frac{1}{f_{\text{sw}} \cdot N} = \frac{1}{100 \times 10^3 \cdot 50}
= 200\,\text{ns}
\end{equation}

At each sub-step, the solver evaluates the state-space equations for the
current switch state (ON/OFF/IDLE), updates the state vector, and checks
for DCM zero-crossing conditions. The simulation runs at approximately
10,000$\times$ real-time on a modern CPU.

\section{7-Metric Evaluation Protocol}

All seven metrics are computed from the output voltage history $V_o(t)$ and
inductor current history $i_L(t)$ over an episode:

\begin{enumerate}[leftmargin=*]
    \item \textbf{Voltage Error}: $| \mathbb{E}[V_o^{\text{steady}}] - V_{\text{ref}} | / V_{\text{ref}} \times 100\%$
    \item \textbf{Voltage Ripple}: $(\max V_o^{\text{steady}} - \min V_o^{\text{steady}}) / V_{\text{ref}} \times 100\%$
    \item \textbf{Efficiency}: $P_{\text{out}} / P_{\text{in}} \times 100\%$, where $P_{\text{out}} = V_o \cdot I_{\text{load}}$, $P_{\text{in}} = V_{\text{in}} \cdot \bar{i_L}$
    \item \textbf{Recovery Time}: Time from last significant disturbance ($>2\times$ ripple band) to return within $\pm2\%$ of $V_{\text{ref}}$
    \item \textbf{Overshoot}: $\max(0, \max V_o - V_{\text{ref}}) / V_{\text{ref}} \times 100\%$
    \item \textbf{Undershoot}: $\max(0, V_{\text{ref}} - \min V_o^{\text{post-startup}}) / V_{\text{ref}} \times 100\%$
    \item \textbf{Startup Time}: Time for $V_o$ to reach 90\% of $V_{\text{ref}}$ and remain within $\pm10\%$ band
\end{enumerate}

The steady-state region is defined as the last 20\% of episode samples.

\section{Compute Environment}

All experiments were run on a single machine with:
\begin{itemize}
    \item CPU: AMD Ryzen 9 7950X (16 cores, 32 threads)
    \item RAM: 64\,GB DDR5
    \item OS: Windows 11, Python 3.11
    \item PyTorch 2.4 (CPU-only, no GPU required)
    \item LLM API: DeepSeek-Chat via OpenAI-compatible endpoint
\end{itemize}

Training time for 200 episodes (200 steps each) was approximately 16--19 minutes
per strategy, with LLM API calls contributing $<$1\% of total runtime.

\section{Test Suite}

The codebase includes 42 automated tests covering:
\begin{itemize}
    \item SAC agent: action selection, update stability, evaluation mode
    \item Replay buffer: push/sample, shape correctness, under-fill protection
    \item CCM/DCM solver: light-load DCM entry, heavy-load CCM, zero-current clamping, mode flag
    \item Boost converter: steady-state convergence, observation shape, reset
    \item Domain randomization: bounds checking, determinism, base-parameter preservation
    \item LLM optimizer: valid JSON output, parameter bounds, weight bounds
    \item Ablation: strategy correctness, vector conversion, GP kernel
    \item Evaluation metrics: steady-state, ripple, efficiency, startup, overshoot, undershoot, recovery
    \item End-to-end trainer: crash-free execution, LLM call at intervention
\end{itemize}

All 42 tests pass (execution time $\sim$4\,s).
